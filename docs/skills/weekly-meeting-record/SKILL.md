# Weekly Meeting Record

用于把会议录音、转写稿、速记内容和审核意见整理成正式会议记录，并按已过审版本的结构与版式输出。

## When To Use

在这些场景主动使用本 skill：

- 用户说要写“会议记录”“例会记录”“会议纪要”。
- 用户给了录音、转写稿、OCR 文本、口语化速记，想整理成正式稿。
- 用户给了上一版被打回的批注，要求你改成过审风格。
- 用户要求“先出 Word，再转 PDF”。
- 用户提到“按之前那几次例会的格式写”“按组织部风格写”。

## Output Rule

始终按两阶段执行：

1. 第一阶段只产出 `.docx` 初稿，不主动转 PDF。
2. 只有在用户明确说“通过了”“可以转 PDF”“导出 PDF”之后，才导出 `.pdf`。

## Workflow

1. 先读取 [`references/style-guide.md`](C:\Users\ZHY\.codex\skills\weekly-meeting-record\references\style-guide.md)。
2. 优先使用转写稿整理内容；如果同时有录音与转写稿，以转写稿为主，录音只用于核对模糊处。
3. 如果用户给了审核截图、批注、语音转述的修改意见，把这些意见视为最高优先级规则，先更新结构和措辞，再出文档。
4. 先做信息补全，再写正文。不要一上来就生成成稿。
5. 从素材中抽取固定字段：
   - 会议标题
   - 日期
   - 星期
   - 开始时间
   - 结束时间
   - 地点
   - 请假人员
   - 迟到人员
   - 旷会人员
   - 主持人
   - 记录人
   - 各板块事项
6. 先完成“内容分组”，再进行“书面改写”。不要一边听一句就塞进一个板块。
7. 如果用户给了“最终过审的 Word/PDF”，把它当成最高优先级模板，优先复用它的分组逻辑和表格版式。
8. 生成结构化 JSON 文件，格式参考 [`assets/sample_input.json`](C:\Users\ZHY\.codex\skills\weekly-meeting-record\assets\sample_input.json)。
9. 使用脚本生成 `.docx`：
   ```powershell
   python C:\Users\ZHY\.codex\skills\weekly-meeting-record\scripts\create_meeting_record.py --input <json路径> --output <docx路径>
   ```
10. 在回复里告诉用户 `.docx` 的绝对路径，并简要说明是否有待确认信息。
11. 只有用户确认初稿通过后，再执行 PDF 导出：
   ```powershell
   powershell -ExecutionPolicy Bypass -File C:\Users\ZHY\.codex\skills\weekly-meeting-record\scripts\convert_docx_to_pdf.ps1 -InputPath <docx路径> -OutputPath <pdf路径>
   ```

## Info Completion Rule

这是新增的硬规则。

- 所有人名相关信息都视为可选输入，不能在 skill 里写死。
- 包括但不限于：
  - 请假人员
  - 迟到人员
  - 旷会人员
  - 主持人
  - 记录人
  - 文中出现的具体负责人姓名
- 在正式出稿前，优先要求用户补全缺失信息。
- 如果用户没有补全，但明确要求“先出初稿”，才允许用 `待确认` 占位。
- 不能因为历史对话里出现过某些固定人名，就自动写入新稿。

## Required Fields Before Final Draft

出正式可审版前，至少确认这些字段：

- 第几次会议
- 日期
- 开始时间
- 结束时间
- 地点
- 请假人员
- 迟到人员
- 旷会人员
- 主持人
- 记录人

如果这些字段缺失：

- 优先向用户索取补全信息
- 若用户只要“先看一版”，可以生成带 `待确认` 占位的初稿
- 但要在回复中明确列出待补全项

## Safe Defaults

下面这些内容可以作为弱默认值使用，但不能覆盖用户补充信息：

- 本学期例会通常在周二下午召开
- 开始时间常见为 `17:10`
- 地点可能固定

注意：

- 这类默认只适用于时间和地点这类非敏感上下文
- 任何人名信息都不属于可自动沿用的默认值

## Writing Rules

遵守以下要求：

- 保持正式、克制、事务性表达，不写口头语、感叹、评价性语言。
- 以“工作事项”和“执行要求”为核心，不写发言过程实录。
- 内容按板块归类，常见板块有：
  - 项目组工作
  - 基层组工作
  - 考评组工作
  - 其他工作
- 若某次会议没有某板块，就不要硬补；按实际内容组织。
- 每条事项写成一句完整的工作要求，优先保留：
  - 时间节点
  - 地点
  - 对象
  - 动作
  - 注意事项
- 人员名单中的“无”保留为“无”，不要改写。
- 标点使用中文全角标点。

## Grouping Rules

这是本 skill 里最容易出错的部分，必须先分组，再成文。

- 优先遵循“该组织已经过审的最终版本”的分组方式，不要用你自己想当然的通用分类法覆盖它。
- 默认可用的大点是：`项目组工作`、`基层组工作`、`考评组工作`、`其他工作`。
- 但默认可用不等于默认必须出现。只有会议里明确提到该板块内容时，才把该大点写进成稿；没提到就不出现。
- `项目组工作` 常见于：活动筹备、宣传转发、微团课、项目执行、会务安排。
- `基层组工作` 常见于：基层团务、副书记例会、转接、团员发展、组织生活会等基层执行事务。
- `考评组工作` 默认是可用板块，但只有素材里确实提到时才写出该大点。
- `其他工作` 默认是可用板块，但只有素材里确实提到无法归入前三类且需要正式保留的事项时才写出该大点。

## Hard Constraints

这几条必须执行：

- 不要把同一主题拆散到多个板块里。先判断它属于哪个组，再整段放进去。
- 不要把“适用于某一部分同学”的要求泛化成“所有同学”。
- 不要把内部口头提醒、默认默契、非正式潜规则写进正式会议记录。
- 不要因为转写里连续说了几段，就自动视为不同板块。板块按事项属性分，不按语音停顿分。
- 不要为了凑满板块数量强行补齐所有板块。
- 不要把历史会议中的具体姓名默认带入本次会议。

## Layout Rules

过审版式按下面执行：

- 标题单独居中，不放进表格。
- 标题下方使用文档自带的表格承载基础信息和正文。不是“看起来像表格”，而是实际的 Word 表格，导出 PDF 后仍保留表格结构。
- 整个正文区域也放在同一张表格中，不要把正文写到表格外。
- 时间行写为：`2026年3月17日星期二 17:10`
- 基础信息区使用表格：
  - 第一行：时间 / 地点
  - 第二行：请假人员
  - 第三行：迟到人员
  - 第四行：旷会人员
  - 第五行：主持人 / 记录人
- 单独一行居中写 `会议内容`
- 正文中必须留空行：
  - `开始时间` 前后各空一行
  - 每个大点之间空一行
  - `结束时间` 前后各空一行
- 正文中每个大点的格式为：
  - `1.项目组工作：`
  - 不要写成带空格的 `1. 项目组工作：`

## Typography Rules

字体和字号按过审成稿执行：

- 标题：`方正小标宋简体`，`18pt`，加粗，居中
- 表格左侧标签列：
  - 字体：`方正小标宋简体`
  - 字号：`12pt`
- 表格内容文字：
  - 字体：`宋体`
  - 字号：`12pt`
- 正文大点标题：
  - 字体：`宋体`
  - 字号：`12pt`
  - 加粗
- `开始时间`、`结束时间` 这两个标签：
  - 标签文字用 `方正小标宋简体`
  - 字号 `12pt`
- 文档页边距优先贴近现成成稿：
  - 上下约 `2.54cm`
  - 左约 `3.18cm`
  - 右约 `3.48cm`

## Review Checklist

出稿前自己过一遍：

1. 板块有没有分错。
2. 有没有凭空多造一个板块。
3. 有没有把局部要求写成全体要求。
4. 有没有把内部口头规则写进正式记录。
5. 文档是不是实际使用了表格，而不只是普通段落排版。
6. `开始时间`、各大点、`结束时间` 之间有没有空行。
7. 缺失的人名信息是否已先向用户索取，或在回复中明确列出待补全项。

## Example Request

“我这有本周例会的录音和转文字，按之前过审版本的风格整理成会议记录，先给我一个 Word 版。缺的考勤和主持信息你先列出来问我补全。”
